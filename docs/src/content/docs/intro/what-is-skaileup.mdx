The **skaileup collection** is the library of *agent skills* the
[`skaile` CLI](https://github.com/skaile-ai/skaile-agent-framework) and the
agent runtime use to drive an LLM through the full product-build pipeline:
problem framing → concept → implementation → quality gates → release.

It is **not** a code library. It is a directory of `SKILL.md` files, plus the
self-contained `flow.yaml` files (each carrying its own dependency manifest) that
compose them, plus the shared `contracts/` reference layer that every skill reads.

## What's in a skill

Every skill is a self-contained directory:

```
my-skill/
├── SKILL.md          ← Required: YAML frontmatter + DSL body (the agent prompt)
├── CLI.md            ← Optional: how to invoke the skill from the CLI
├── references/       ← Optional: detailed docs the body links to
├── scripts/          ← Optional: helper executables
├── tests/            ← Optional: test manifest + fixtures for `lab/validate`
└── validator.py      ← Optional: compiled MUST/NEVER checker (`lab/compile-validators`)
```

The agent loads the SKILL.md frontmatter at startup (~100 tokens per skill for
`name + description`) and reads the body only when the skill activates
(under 5000 tokens). Heavier reference material lives in `references/` and is
loaded on demand. This is the
[**progressive-disclosure**](https://agentskills.io/specification) pattern that
keeps the agent's context window cheap.

## Who consumes it

| Consumer | How |
|---|---|
| `skaile` CLI | Walks the repo, builds a catalog from manifest filenames (`SKILL.md`, `*.flow.yaml`, etc.), installs to `.claude/skills/` (or `.omp/`, `.codex/`) |
| Skaile platform | Reads SKILL.md files from disk at runtime; routes the agent's `Skill` tool to them |
| Custom agents | Any agent that supports the [agentskills.io spec](https://agentskills.io/specification) |

The repo is mounted as a **git submodule at `ai-assets-skaileup/`** in the
skaile-dev shell repo, alongside `ai-assets/`. Local checkouts use `path:` in
`skaile.yaml` to symlink (live updates); remote checkouts use `url:` and copy.

## Why this collection exists

LLMs are good at writing code one file at a time and bad at carrying
project-shaped intent across long sessions. The skaileup collection encodes that
intent as discrete, verifiable steps:

- **Slice it.** Big apps are built one feature at a time, not all twenty
  features upfront.
- **Reset context.** Every slice phase reads from the prior phase's scratch
  file, not from chat history. Run `/clear` between phases.
- **Gate the work.** Iron laws (no concept work without a brief; no screens
  without a data model) sit in `contracts/` and every skill enforces them.
- **Verify mechanically.** `MUST` / `NEVER` / `CHECKLIST` blocks compile down to
  Python validators — sub-second checks instead of a second LLM round-trip.

The collection is the **operating system** for an agent that is supposed to ship
software, not the agent itself.
