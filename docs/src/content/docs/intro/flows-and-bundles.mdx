A **flow** is a YAML graph of skill nodes and edges — a runnable path through the
skill tree. A **bundle** is an **installation manifest**: the list of skills that
flow needs installed. They come in pairs, co-located under `skaileup/flows/<tier>/`:

```
skaileup/flows/
├── concept-slice/
│   ├── concept-slice.flow.yaml
│   └── concept-slice.bundle.yaml
├── impl-slice/
│   ├── impl-slice.flow.yaml
│   └── impl-slice.bundle.yaml
├── mvp/
│   ├── mvp.flow.yaml
│   └── mvp.bundle.yaml
├── simple-app/
│   ├── simple-app.flow.yaml
│   └── simple-app.bundle.yaml
├── standard-app/
│   ├── standard-app.flow.yaml
│   └── standard-app.bundle.yaml
└── complex-app/
    ├── complex-app.flow.yaml
    └── complex-app.bundle.yaml
```

## Install — bundles are subsets

Bundles install skills via the **skaile workspace CLI**. Install the whole
collection, or only the subset a flow needs:

```bash
$ skaile add skill:*                    # install every skill
$ skaile add bundle:simple-app          # OR install only what simple-app needs
```

A bundle never *runs* anything — it only decides which skills land in the workspace.

## Run — engine or orchestrator

Once the skills are installed, a flow can be run **two interchangeable ways**:

```bash
$ skaile run flow:simple-app            # 1. the skaile workspace flow engine (connector)
```

```text
2. the orchestrator (skaileup / skaileup-build) reads the same .flow.yaml and
   guides/executes it conversationally — used when there's no flow engine, or
   when you want a human-in-the-loop run.
```

Both follow the identical node graph; pick the engine for hands-off execution,
the orchestrator for a guided run. Either way, the flow's skills must already be
installed (whole collection, or the flow's bundle).

The two slice flows (`concept-slice`, `impl-slice`) are **building blocks**.
Tier flows compose them.

```mermaid

    direction LR
    s1[scope-project] --> s2[linear concept] --> s3[impl-build setup] --> s4["loop: impl-slice"] --> s5([done — all features built])

```

```mermaid
flowchart LR
  subgraph standard["standard-app.flow.yaml"]
    direction LR
    t1[scope-project] --> t2["high-level concept<br/>brief · brand · journeys"] --> t3[impl-build setup] --> t4["loop: concept-slice → impl-slice"] --> t5([done])
  end
```

```mermaid
flowchart LR
  subgraph complex["complex-app.flow.yaml"]
    direction LR
    c1[scope-project] --> c2[high-level concept] --> c3[project-overview] --> c4[impl-build setup] --> c5["loop: concept-slice → impl-slice<br/>supervised plan + impl-quality/audit every slice"] --> c6([done])
  end
```

Bundles inherit: `simple-app` includes `mvp`, `standard-app` includes
`simple-app`, `complex-app` includes `standard-app`. Each bundle file lists
only its *additions*.

## Bundle-from-flow generation

`lab/compile-bundle` walks a flow's node graph and emits the matching
`<name>.bundle.yaml`. **Run on every flow change** to prevent drift between
what the flow executes and what the bundle installs. The repo also ships a
`scripts/check-bundles.sh` that enforces this in CI.

## Custom flows

You can create your own flow by starting from a tier and overriding what you need. Create a file such as `skaileup/flows/custom/custom.flow.yaml`:

```yaml
extends: standard-app
nodes:
  - id: my-extra-step
    type: skill
    data:
      skill: ops/sync
      after: [impl-slice]
```
