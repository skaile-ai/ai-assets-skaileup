## Authoritative spec excerpt (from REFACTOR_MOCKUP.md § 6)

The contract being authored MUST capture the following exactly:

```yaml
---
implements:
  - experience/features/01_user_auth/login.md
data_entities: [User]
layout: experience/screens/00_layout/shell.md
elements:                                  # NEW — optional
  - id: email-input
    kind: input
    label: "Email"
    states: [default, focus, error]
  - id: password-input
    kind: input
    label: "Password"
    states: [default, focus, error]
  - id: submit-button
    kind: button
    label: "Sign in"
    states: [default, loading, disabled, error]
last_updated: 2026-05-06
---
```

Walkthrough renderers MUST emit:
- `data-spec-screen="<screen-path>"` on the screen root
- `data-spec-element="<element-id>"` on every annotatable node
- `data-spec-provisional="true"` when the ID was auto-slugged (no explicit `elements:` entry)

**Hybrid ID strategy** (verbatim from § 6):
1. Initial render: walkthrough auto-slugs IDs from labels/text. Marks all IDs `provisional`.
2. First annotation on a provisional element: `mockup-feedback-triage` prompts to promote the ID to explicit. The promoted ID gets written into the screen's `elements:` frontmatter via patch.
3. Subsequent renders use the promoted ID, no longer provisional. Future regeneration of the screen preserves the ID.

This avoids upfront tedium AND ID instability across regenerations.

## Step ordering rule

Execute strictly in this order; do NOT reorder:

1. **Contract first** (`contracts/elements_block.md`) — the schema is the source of truth that everything else is checked against.
2. **Examples second** (`tests/elements_block_examples.md`) — three valid + three invalid examples written from the contract.
3. **Validator third** (`lab/validate-elements-block/SKILL.md` + `validator.py`) — TDD: examples exist before validator, validator developed against them.
4. **Cross-link last** (`contracts/frontmatter.md` extension) — only after the contract is stable do we publish the pointer.

The reason: it lets us TDD the validator. We can never end up with a validator whose rules disagree with the contract, because the validator is written *to make the prepared examples pass/fail*.

  Use the SKILL.md DSL from `contracts/skill_grammar.md`. Frontmatter (per `contracts/asset_frontmatter.md` § Skill):

  ```yaml
  ---
  name: lab-validate-elements-block
  description: "Use when validating the `elements:` block in screen frontmatter or example fixtures. Returns 0 for valid, non-zero with line numbers for invalid."
  metadata:
    version: "0.1.0"
    tags: [validation, elements, frontmatter, screens, lab]
    stage: alpha
    requires:
      - contract:elements-block
    prerequisites:
      reads:
        - path: "contracts/elements_block.md"
          description: "Schema being validated against"
        - path: "tests/elements_block_examples.md"
          description: "Default fixture set if no target supplied"
      produces:
        - path: "stdout"
          description: "Pass/fail report with file:line for each violation"
  ---
  ```

  Body (skill DSL):

  - **ROLE** — one-sentence: validates `elements:` blocks in YAML frontmatter against the schema in `contracts/elements_block.md`.
  - **READS** — `tests/elements_block_examples.md` (default), or any `experience/screens/**/*.md` passed as argument.
  - **WRITES** — stdout (no files).
  - **REFERENCES** — `contracts/elements_block.md`, `contracts/scripts/validator_lib.py`.
  - **MUST** — report file:line for every invalid element. Exit 0 if all examples match their declared `expect:`.
  - **NEVER** — call an LLM. Validation is purely structural.
  - **STEP 1** — Resolve target (CLI arg or default fixture).
  - **STEP 2** — For each example block (extracting via the `<!-- example: ... -->` sentinel), parse YAML, run schema checks, compare result to declared `expect:`.
  - **STEP 3** — Print summary; exit 0 iff all expectations match.
  - **CHECKLIST** — file resolves, all 6 examples discovered (in fixture mode), report is line-numbered.

- [ ] **Step 3.5: Author `lab/validate-elements-block/validator.py`.**

  Reuse `contracts/scripts/validator_lib.py` for the report-formatting plumbing (Validator class, exit-code wiring via `main`). The custom logic is:

  1. **Parse argv:** if `len(sys.argv) > 1`, target = `sys.argv[1]`; else default = `tests/elements_block_examples.md`.
  2. **If target is a fixture file** (contains `<!-- example: ... -->` sentinels): extract each example block, record its declared `expect:` and the line range it occupies in the source file, parse the YAML inside, run `_validate_elements(parsed)`, and assert result matches `expect:`.
  3. **If target is a screen markdown file** (`experience/screens/**/*.md`): parse YAML frontmatter; if `elements:` is absent, exit 0 (the field is optional); else run `_validate_elements(fm['elements'])`.
  4. **`_validate_elements(elements)`** returns a list of `(line_offset, message)` violations. Checks:
     - `elements` is a list (not a mapping or scalar).
     - Each entry is a mapping.
     - Each entry has required keys `id`, `kind`, `label`, `states`.
     - `id` is kebab-case, matches `^[a-z][a-z0-9-]*[a-z0-9]$` and contains no `--`.
     - `kind` is in the enum (mirror the enum from `contracts/elements_block.md`).
     - `states` is a non-empty list of strings; each value is in the states enum (mirror enum).
     - `ids` are unique within the block.
     - Optional fields, if present, have correct types: `provisional: bool`, `describes: str`, `data_entity: str`, `acceptance_refs: list[str]`.
  5. **Output format:** for each violation print `<path>:<line>: <message>` (line number relative to the source file, not relative to the YAML block — compute by adding the block start offset).
  6. **Exit code:** 0 if all checks pass (or all expectations are satisfied in fixture mode); 1 otherwise.

  Reference the existing `experience/screens/validator.py` for the `Validator` boilerplate. The custom YAML/regex logic is local to this validator; the `Validator` class is used to format the final report.

## Open Questions / Ambiguities

These were surfaced during planning rather than guessed. The executing agent SHOULD confirm or escalate before locking in:

1. **`kind` enum and `states` enum.** REFACTOR_MOCKUP.md § 6 shows examples (`input`, `button`, `default`, `focus`, `error`, `loading`, `disabled`) but does NOT publish a closed enum. The mini-plan proposes:
   - `kind`: `input`, `button`, `link`, `image`, `text`, `region`, `list`, `form`, `nav`, `media`, `custom`.
   - `states`: `default`, `focus`, `hover`, `active`, `disabled`, `loading`, `error`, `success`, `empty`.

   These are reasonable defaults but should be confirmed. The contract should mark them as "v0.1 — open for extension" so future renderer skills can propose additions without being blocked.

2. **`asset_frontmatter.md` vs `frontmatter.md`.** The parent plan stub (Task 2.0) says: *"Modify: `contracts/asset_frontmatter.md` (extend the screen frontmatter section to reference `elements_block.md`)"*. But screen frontmatter is documented in `contracts/frontmatter.md`, NOT `asset_frontmatter.md` — the latter covers SKILL.md / AGENT.md / etc. manifest schemas. Recommended interpretation: extend `frontmatter.md` (the semantically correct file) AND add a one-line pointer in `asset_frontmatter.md` to satisfy the literal instruction. This is what Tasks 4.1 and 4.2 do. Flag for human review if a different intent was meant.

3. **`elements: []` vs unset.** The contract should specify whether an empty list and an absent key are semantically identical (both "no elements declared, fall back to auto-slug"). The mini-plan assumes yes; confirm before publishing.

4. **`acceptance_refs` format.** Proposed format `<feature-path>#<criterion-id>` mirrors the existing `story_refs:` pattern in `contracts/frontmatter.md`. If the project uses a different anchor convention, swap.

5. **Validator distribution.** The validator skill lives in `lab/`. Whether it should also be invocable as a hook (like `contracts/scripts/validate_skill_rules.py`) is out of scope for Task 2.0 but worth noting for Phase 3.

6. **Test file location.** `tests/` does not currently exist at the repo root. The mini-plan creates it. If the project prefers `lab/validate-elements-block/examples/` (skill-local), the executing agent should confirm. The parent plan stub explicitly says `tests/elements_block_examples.md`, so the mini-plan follows that.
