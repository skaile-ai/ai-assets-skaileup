# Semantic Type System

The data model uses stack-independent semantic types.
These types carry **intent** — a stack translator maps them to concrete output formats.

Skills always write semantic types in `model.json` and `model.dbml`.
Stack-specific output (Prisma schema, postxl-schema.json, SQL DDL) is produced by
the datamodel skill as a translation step, driven by the detected tech stack.

## Scalar Types

| Type | Meaning | Example fields |
|---|---|---|
| `string` | Short text (≤ 255 chars) | name, title, slug, email, url |
| `text` | Long plain text | description, bio, notes |
| `richtext` | Formatted content (HTML/markdown) | article body, changelog |
| `integer` | Whole number | count, sort_order, age |
| `float` | Decimal number | rating, latitude |
| `decimal` | Precise decimal (financial, scientific) | amount, balance, tax_rate |
| `boolean` | True/false | is_active, is_published |
| `date` | Date only (no time) | birth_date, due_date |
| `datetime` | Date + time with timezone | created_at, published_at |
| `json` | Arbitrary structured data | settings, metadata, preferences |
| `uuid` | Unique identifier | id, external_id |
| `password` | Secret (hashed at app layer) | password_hash |
| `image` | Image file reference | avatar, cover_image |
| `file` | Any file reference | attachment, document |

## Optionality

Every field is required by default. Mark a field optional with `required: false`.
The `?` suffix used in Prisma/PostXL output is a translation artifact — do not write it in semantic models.

## Enum Type

Enums are defined inline as an object mapping **PascalCase** values to descriptions:

```json
{
  "name": "status",
  "type": "enum",
  "values": {
    "Draft": "Initial state, not yet submitted",
    "InReview": "Awaiting approval",
    "Published": "Live and visible"
  }
}
```

- Enum values are always PascalCase (never `SCREAMING_SNAKE` or `lowercase`)
- Shared enums reused across models are defined at the top level of the schema and referenced by name
- The stack translator generates the concrete enum definition (Prisma `enum`, SQL `CHECK`, etc.)

## Relation Types

| Type | Meaning | Example |
|---|---|---|
| `m2o` | Many-to-one | task.assigned_to → User |
| `o2m` | One-to-many (inverse of m2o) | user.tasks → Task[] |
| `m2m` | Many-to-many | Task ↔ Tag (via junction model) |

**Rules:**
- `o2m` inverses are auto-generated by the stack translator — do not declare them manually
- `m2m` requires an explicit junction model in the semantic schema (e.g. `TaskTag` with `m2o` to both sides)
- Every `m2o` field name ends with `_id` (semantic layer uses snake_case)

## Model Metadata

Every model supports these metadata hints, consumed by the stack translator and UI scaffolding:

| Property | Purpose |
|---|---|
| `label_field` | Field used as display name in UI lists and dropdowns |
| `key_field` | URL-friendly unique identifier field (slug / readable URL) |
| `standard_fields` | Auto-generated fields to include: `id`, `created_at`, `updated_at` |
| `faker` | Per-field hint for seed data generation (e.g. `"faker.internet.email"`) |

`standard_fields` defaults to `["id", "created_at", "updated_at"]` for every model.
Do **not** declare these manually in the fields list — they are injected by the translator.

## Field Properties

All fields support these optional properties:

| Property | Type | Purpose |
|---|---|---|
| `required` | boolean | Whether the field must have a value (default: `true`) |
| `unique` | boolean | Enforce uniqueness constraint |
| `readonly` | boolean | Prevent editing after creation |
| `indexed` | boolean | Create a database index |
| `max_length` | number | Maximum character length (string fields) |
| `placeholder` | string | UI hint text |
| `faker` | string | Seed data generator hint |

## Rules for Skills

1. **Always use semantic types** in `model.json` and `model.dbml` — never raw SQL or Prisma types
2. **Model names** are PascalCase singular (`Task`, `ProjectMember`)
3. **Field names** are snake_case (`first_name`, `is_active`, `project_id`)
4. **Every model** declares `standard_fields: ["id", "created_at", "updated_at"]` — do not add these to the fields array
5. **Enum values** are PascalCase (`Draft`, `InProgress`, `Done`)
6. **Every model** sets `label_field` so the UI knows which field to display
7. **Set `key_field`** when the model has a human-readable unique slug
8. **Relations**: use `m2o` + `_id` suffix for the owning side; declare junction models explicitly for `m2m`; never declare `o2m` — it is generated
9. **Stack-specific concerns** (Prisma decorators, Directus system collections, PostXL `standardFields` casing) are handled by the stack translator, not by the datamodel skill
