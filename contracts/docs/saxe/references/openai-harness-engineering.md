# Harness Engineering: Leveraging Codex in an Agent-First World

- Author: Ryan Lopopolo (OpenAI)
- Date: February 11, 2026
- Source: https://openai.com/index/harness-engineering/

## TL;DR

OpenAI describes an internal experiment where a product was built with no
manually written code: Codex generated app code, tests, CI, docs, tooling, and
operational artifacts. Humans focused on steering, constraints, and feedback
systems. The reported result is much higher throughput, with an explicit need
for strong architecture controls and continuous cleanup to prevent drift.

## Context and Reported Outcomes

- Project started from an empty repo in late August 2025.
- Initial scaffold (repo structure, CI, framework setup, formatting rules,
  AGENTS instructions) was generated by Codex.
- After roughly 5 months:
  - About 1M lines of code.
  - About 1,500 merged PRs.
  - Team grew from 3 to 7 engineers.
  - Throughput around 3.5 PRs per engineer per day and increasing.
- Product was used internally and by external alpha testers.
- Working principle: humans do not hand-write code; agents produce code and
  related artifacts.

## Role Shift: From Coding to Harness Engineering

The human role moves up a level:

- Design environments where agents can succeed.
- Break goals into constrained, testable tasks.
- Encode constraints and acceptance criteria.
- Build recovery loops and escalation paths.
- Add missing capabilities when failure patterns appear.

Typical loop described:

1. Human prompts task.
2. Agent implements and opens PR.
3. Agent self-reviews and receives agent review.
4. Feedback is iterated.
5. Human escalates only where judgment is needed.

## Making Systems Legible to Agents (App + Ops)

Main bottleneck became human QA, so they expanded what agents can validate:

- Per-worktree app instances to run each change in isolation.
- Chrome DevTools Protocol automation for UI checks.
- Local ephemeral observability stack available to agents:
  - Logs (LogQL)
  - Metrics (PromQL)
  - Traces
- Agents validate behavior, performance, and reliability targets directly.
- Single autonomous runs can continue for many hours.

## Documentation Strategy: Map, Then Depth

OpenAI reports that a single giant `AGENTS.md` did not scale.

Their pattern:

- Keep `AGENTS.md` short, acting as a map/table of contents.
- Store substantive knowledge in versioned docs under repo folders.
- Include design docs, product specs, plans, debt trackers, quality docs, and
  references as first-class assets.
- Add CI/linting rules that enforce freshness and structure.
- Run recurring doc-maintenance agent tasks.

Core principle: progressive disclosure. Start with a compact map, then load
deeper context only when needed.

## Agent Legibility as a Primary Design Goal

If important context is not in-repo, an agent cannot reliably use it.

Implications:

- Move key decisions out of chat/history and into versioned artifacts.
- Prefer abstractions and tools agents can reason about deterministically.
- In some cases, reimplement small internal utilities if third-party libraries
  are too opaque.
- Optimize for long-term machine navigability, not only current human fluency.

## Enforcing Architecture and Quality Mechanically

Written conventions were not enough, so constraints were enforced by tooling:

- Domain layering with strict dependency direction.
- Clear boundary points for cross-cutting concerns.
- Custom linters and structural tests for architecture compliance.
- Additional invariants (examples): structured logging, naming standards,
  file-size limits, reliability constraints.
- Lint errors include remediation hints tailored for agents.

Operating idea:

- Strong central constraints.
- Local autonomy within constraints.
- Correctness, maintainability, and legibility over style preference.

## Throughput Changes Merge Philosophy

With high agent throughput and scarce human attention:

- Keep PRs short-lived.
- Minimize blocking gates where possible.
- Fix flakes and minor regressions quickly in follow-up runs.
- Prefer continuous correction over long queue delays.

The article also notes this tradeoff is context-dependent and can be risky in
low-throughput settings.

## What "Agent-Generated" Included

Agents generated more than product code:

- Application code and tests.
- CI and release configuration.
- Internal developer tooling.
- Documentation and design history.
- Evaluation harnesses.
- Review comments and responses.
- Repository management scripts.
- Dashboard definitions.

Humans remained responsible for prioritization and acceptance.

## Increasing End-to-End Autonomy

With enough instrumentation and constraints, a single prompt can trigger:

- Validation of current state.
- Bug reproduction and evidence capture.
- Fix implementation.
- Revalidation and evidence update.
- PR creation and iterative review handling.
- Build break repair and merge, with escalation only for judgment calls.

OpenAI frames this as repository-specific, not universally automatic.

## Entropy and Continuous Garbage Collection

Autonomous systems amplify both good and bad local patterns.

Their approach evolved from periodic cleanup to continuous maintenance:

- Define "golden principles" as enforceable rules.
- Run recurring cleanup and refactor tasks.
- Track quality continuously.
- Repay debt in small increments.

Framing: technical debt behaves like high-interest debt; continuous repayment is
more effective than occasional large cleanups.

## Open Questions Highlighted

- Can long-term architectural coherence be sustained over multiple years?
- Where does human judgment have the highest leverage?
- How can human judgment be encoded so it compounds?
- How should workflows change as models improve?

## Skill-Design Implications for This Repo

Use these as direct inputs when adding or refining skills:

1. Optimize for agent legibility.
2. Keep top-level instructions short and map-like; link to structured depth.
3. Enforce architecture and contracts through automated checks.
4. Treat plans, decisions, and debt records as first-class artifacts.
5. Provide closed validation loops agents can run directly (UI + logs + metrics
   - traces where relevant).
6. Add recurring cleanup tasks so quality does not drift.
