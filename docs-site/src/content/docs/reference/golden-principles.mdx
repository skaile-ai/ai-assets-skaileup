## Entity Rules

- Every entity MUST declare `standard_fields` including at least `["id", "created_at", "updated_at"]`
- The `id` field MUST have type `uuid` and `primary: true`
- `created_at` and `updated_at` MUST have type `datetime`
- Entity names are `lowercase_snake_case` singular: `user`, `task`, `project_member`
- No two entities may share the same name

> **Translation note:** Output conventions (PascalCase model names in
> Prisma/PostXL, camelCase fields) are applied by the stack translator —
> never in the semantic layer.

## Field Rules

- All field names are `lowercase_snake_case` in the semantic layer: `display_name`, `assigned_to_id`
- Field type is a semantic type (see `semantic_types.md`) — never a backend-specific type
- Optional fields use `required: false` — not a `?` suffix (translation artifact)

## Enum Rules

- Enum values are PascalCase: `Draft`, `InProgress`, `Done`
- Inline enums use object format: `{"Draft": "description", "InProgress": "description"}`
- Inline enums auto-name as `{entity_name}_{field_name}` (snake_case): `task_status`, `user_role`
- Every named enum must be referenced by at least one field
- Standalone enum IDs are `lowercase_snake_case` matching their purpose

## Relation Rules

- Many-to-one (`m2o`) field names MUST end with `_id`: `owner_id`, `project_id`
- The field type is the referenced entity name (snake_case): `user`, `project`
- Inverse relations are auto-generated by the translator — do not declare them
- Many-to-many (`m2m`) relations require an explicit junction entity

## Feature Rules

- Every feature file MUST have at least one requirement checkbox (`- [ ]`)
- Every feature file MUST have a `## Description` section with content
- `priority` must be `must-have` or `nice-to-have` — no other values
- `roles` must be a non-empty array

## Feature Group Rules

- Group folders use two-digit numbered prefixes: `01_`, `02_`, `03_`
- Numbers are sequential with no gaps (`01_`, `03_` without `02_` is invalid)
- Group names are `lowercase_snake_case`: `01_user_auth`, not `01_UserAuth`
- Feature group numbers mirror screen group numbers exactly

## Screen Rules

- Every screen MUST have `implements:` with at least one feature path
- Every screen MUST have a `## Component Inventory` section with at least one item
- Every screen MUST have a `## States` section (at minimum: default, loading, error)
- Screen groups mirror feature group numbers exactly

## Cross-Reference Rules

- Every path in frontmatter is relative to `_concept/`
- Every path in frontmatter MUST resolve to an existing file or folder
- If a screen lists a feature in `implements:`, that feature MUST list the screen in `screens:`
- If a feature has `data_entities:`, each entity MUST exist in `model.json`
- If a feature has `story_refs:`, each referenced story ID MUST exist in `stories.json`

> `model.json` is the canonical stack-agnostic data model. Stack-specific
> files (`schema.prisma`, `postxl-schema.json`) are derived — cross-references
> always target `model.json`.

## Frontmatter Rules

- Every `.md` file in `_concept/` MUST have YAML frontmatter
- Every frontmatter MUST include `last_updated`
- `last_updated` must be a valid ISO date (`YYYY-MM-DD`)

## Naming Rules

- All filenames are `lowercase_snake_case` with `.md` extension
- No spaces in any path under `_concept/`
- No special characters beyond underscore (`_`) and hyphen (`-`)

## Seed Data Rules

- `seed.json` MUST include at minimum four named scenarios: `empty`, `single_user`, `populated`, `edge_cases`
- The `populated` scenario MUST contain at least 2 entries per entity
- The `edge_cases` scenario MUST include: long names, special characters, empty optional fields
- All field names in seed data are `lowercase_snake_case` matching the semantic model
- Enum values in seed data are PascalCase matching the inline enum definitions
- Relation fields (`*_id`) MUST reference valid IDs from the related entity's entries in the same scenario
- Data must be realistic: real-looking names, varied lengths, mixed locales
